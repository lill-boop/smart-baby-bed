### 7.2.2 AI 姿态识别与预警准确性测试

**测试目标：** 验证 YOLOv8-Pose 模型在 RKNN NPU 加速下的检测精度与实时性。

**1. 模型推理性能测试**

使用 `test_rknn.py` 脚本对量化后的 RKNN 模型进行基准测试：

```python
#!/usr/bin/env python3
from rknnlite.api import RKNNLite
import cv2
import numpy as np

rknn = RKNNLite()
rknn.load_rknn('/var/log.hdd/babycam/ai/yolov8n-pose.rknn')
rknn.init_runtime()

# 从 RTSP 流获取测试帧
cap = cv2.VideoCapture('rtsp://127.0.0.1:8554/babycam')
for _ in range(5): cap.grab()
ret, frame = cap.read()
cap.release()

# 预处理（NCHW 格式，UINT8 保持 0-255）
img = cv2.resize(frame, (640, 640))
img = cv2.cvtColor(img, cv2.COLOR_BGR2RGB)
img = np.transpose(img, (2, 0, 1))  # HWC -> CHW
img = np.expand_dims(img, axis=0)   # CHW -> NCHW

# NPU 推理
outputs = rknn.inference(inputs=[img], data_format='nchw')
output = outputs[0][0].T

# 置信度分析
scores = output[:, 4]
print(f'检测到的目标数: {(scores > 0.5).sum()}')
print(f'最高置信度: {scores.max():.4f}')

rknn.release()
```

### 7.2.3 视频流传输与延迟测试

**测试方法：** 采用"秒表拍摄法"（Glass-to-Glass Latency），在摄像头前放置运行的数字秒表，同时拍摄手机屏幕显示的画面，计算时间差。

**测试结果：**

| 视频流类型 | 编码格式 | 分辨率 | 帧率 | 端到端延迟 | 结论 |
|-----------|---------|-------|------|-----------|------|
| 原始 RTSP 流 | H.264 | 1280x720 | 30 fps | 380-480ms | ✅ 优秀 |
| AI 分析 MJPEG 流 | MJPEG | 640x480 | 20 fps | 220-280ms | ✅ 优秀 |

**延迟来源分析：**

```
总延迟 = 采集延迟 + 编码延迟 + 网络传输 + 解码延迟 + 渲染延迟
        (33ms)   + (80ms)   + (150ms)  + (120ms)  + (97ms)
        = 480ms
```

**优化措施验证：**
- **FFmpeg `-tune zerolatency`**：禁用 B 帧，降低缓冲延迟 ~100ms
- **ExoPlayer 自定义 LoadControl**：缩减播放缓冲至 100-500ms 范围
- **RTSP over TCP**：避免 UDP 丢包重传导致的画面卡顿

### 7.2.4 事件触发录像与回溯测试

**测试流程：**
1. 模拟哭声事件，STM32 发布 MQTT 触发消息
2. Orange Pi 收到消息，调用 `export_clip_remote.py` 脚本
3. 从环形缓冲区提取前 10s 至后 20s 共 30s 视频片段
4. FFmpeg 合成 MP4 文件并推送至 App 事件列表
5. 用户点击回放，验证音视频同步

**export_clip_remote.py 核心逻辑：**
```python
def newest_segments(seconds: int) -> list[Path]:
    """从环形缓冲区中获取最近 N 秒的有效 TS 切片"""
    now = time.time()
    segs = []
    for p in RING.glob("*.ts"):
        st = p.stat()
        if st.st_size > 0 and (now - st.st_mtime) <= seconds + 5:
            segs.append((st.st_mtime, p))
    segs.sort(key=lambda x: x[0])
    return [p for _, p in segs]

def export_from_list(files: list[Path], out_path: Path):
    """使用 FFmpeg concat 协议合成 MP4"""
    list_path = TMPD / "list.txt"
    with list_path.open("w") as w:
        for f in files:
            w.write(f"file '{str(f)}'\\n")
    
    # 尝试直接 copy codec（速度快）
    cmd = ["ffmpeg", "-y", "-f", "concat", "-safe", "0", 
           "-i", str(list_path), "-c", "copy", str(out_path)]
    subprocess.run(cmd, check=True)
```

**测试结果：**

| 测试项 | 指标 | 实测值 | 结论 |
|-------|------|-------|------|
| 触发响应时间 | MQTT 消息到脚本启动 | 120ms | ✅ 实时 |
| 视频合成耗时 | 30s 片段（直接 copy） | 2.3s | ✅ 高效 |
| 文件大小 | H.264 1080P 30s | ~8MB | ✅ 合理 |
| App 加载时间 | 从列表点击到播放 | 0.8s | ✅ 流畅 |
| 音视频同步 | A/V 偏移 | < 50ms | ✅ 优秀 |

## 7.3 系统性能与鲁棒性分析

### 7.3.1 NPU 加速性能对比分析

**对比实验设置：** 使用相同的 YOLOv8n-Pose 模型，分别在 CPU 和 NPU 上运行推理，输入 1080P 视频流。

**性能对比表：**

| 运行模式 | 硬件资源 | 推理延迟 | 帧率 | CPU 占用 | 功耗 |
|---------|---------|---------|------|---------|------|
| **CPU 软推理** | Cortex-A76 4核 | 125ms | 8 FPS | 85% | 4.2W |
| **NPU 硬件加速** | RK3588 NPU 6 TOPS | 28ms | 35 FPS | 12% | 2.8W |
| **性能提升** | - | **↓ 77%** | **↑ 4.4x** | **↓ 73%** | **↓ 33%** |

**资源占用细节：**
- **内存占用**：NPU 模式下系统内存占用稳定在 680MB，无内存泄漏
- **温度控制**：持续推理 1 小时，NPU 核心温度稳定在 52-58℃，未触发降频
- **并发能力**：NPU 承担推理任务后，剩余 CPU 核心可同时处理 3 路 H.264 编码

## 7.4 测试总结与问题分析

### 7.4.1 测试结论

通过系统化的功能与性能测试，验证了本智能婴儿床监控系统在以下方面达到设计目标：

**✅ 功能完整性：**
- 环境感知精度：温度 ±0.2℃，湿度 ±3%
- 自动控制响应：平均延迟 < 1s
- AI 姿态识别：综合准确率 96.9%
- 事件录像回溯：100% 成功率

**✅ 性能优越性：**
- NPU 加速推理：35 FPS，延迟 28ms
- 端到端延迟：RTSP 流 380ms，AI 流 240ms
- MQTT 通信：QoS 1 消息成功率 99.2%

**✅ 稳定性与鲁棒性：**
- 72 小时连续运行无崩溃
- 网络断连自动恢复时间 < 10s
- 异常容错机制完善

### 7.4.2 发现的问题与改进方向

**1. 弱光环境识别率下降**
- **现象**：光照 < 5 Lux 时，人体关键点检测置信度降低 15%
- **原因**：训练集缺乏低照度样本
- **改进**：增加红外补光灯或训练夜视模式模型

**2. 多人入镜误判**
- **现象**：家长进入画面时，系统偶尔将成人识别为主目标
- **改进**：增加基于身高比例的人体尺寸过滤逻辑

**3. 长时间录像存储压力**
- **现象**：24 小时环形缓冲占用约 20GB 存储
- **改进**：引入动态码率调整或 H.265 编码降低存储需求
